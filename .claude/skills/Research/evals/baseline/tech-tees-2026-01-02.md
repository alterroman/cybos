## Technology Overview

### What Are TEEs?

Trusted Execution Environments are **hardware-enforced isolated computing environments** that protect code and data from access by privileged software (including operating systems, hypervisors, and cloud administrators). They create a "secure enclave" within a processor where:

- **Code executes in isolation**: Even root access cannot inspect enclave memory
- **Data remains encrypted in memory**: Only decrypted inside the enclave using hardware-protected keys
- **Remote attestation proves integrity**: Third parties can cryptographically verify code running in enclave matches expected hash

### Architecture & How TEEs Work

TEEs operate through three core mechanisms:

**1. Memory Encryption Engine (MEE)**
- Hardware encrypts all memory pages belonging to enclave using AES-128/256
- Encryption keys stored in CPU-internal fuses, never exposed to software
- Memory integrity verified via Merkle trees to prevent tampering

**2. Isolation & Access Control**
- CPU enforces page-table restrictions preventing non-enclave code from accessing enclave memory
- Interrupts and exceptions are handled without exposing enclave state
- Side-channel protections (cache partitioning, memory scrambling)

**3. Remote Attestation**
- CPU generates signed attestation report containing:
  - Hash of enclave code and initial data
  - Platform security version (firmware/microcode levels)
  - Additional user data (e.g., public keys)
- Report signed by hardware-protected attestation key
- Verifier checks signature against manufacturer's public key

### Four Production Implementations

| Technology | Vendor | Isolation Level | TCB Size | Performance | Use Case Fit |
|------------|--------|----------------|----------|-------------|--------------|
| **SGX** | Intel | Process-level enclaves | ~1MB | 3-15% overhead | Legacy, small workloads |
| **SEV-SNP** | AMD | Full VM encryption | ~50MB | 5-10% overhead | **Best price/performance** |
| **TDX** | Intel | Full VM (Trust Domains) | ~10MB | 8-15% overhead | **Smallest TCB, newest** |
| **H100 TEE** | NVIDIA | GPU-accelerated confidential | ~100MB | 10-15% overhead | **Confidential AI inference** |

**Key Architectural Difference**: SGX isolates individual processes (small enclaves, larger attack surface). SEV-SNP/TDX/H100 isolate entire VMs (larger memory footprint, simpler programming model, production-ready for cloud workloads).

### Critical Performance Benchmark: The "1-Second Auction Test"

Serokell's investment thesis requires privacy tech to support real-time use cases (MEV auctions, programmatic ad bidding, high-frequency trading):

- **TEEs**: 3-15% overhead → **1-second auction becomes 1.15 seconds** ✅ PASS
- **FHE (Fully Homomorphic Encryption)**: 1000x overhead → **1-second auction becomes 16 minutes** ❌ FAIL

This performance gap is **structural, not incremental**. FHE requires computation on encrypted data (extreme overhead). TEEs decrypt inside hardware-protected enclave (minimal overhead). No software optimization will close this 100x-1000x gap.

**Implication**: TEEs enable real-time confidential computing. FHE remains limited to batch/asynchronous workloads (medical research, privacy-preserving analytics where latency tolerance is hours/days).

### Known Limitations & Attack Vectors

**1. Side-Channel Attacks (Historical Vulnerability)**
- **SGX compromises**: Spectre/Meltdown (2018), Plundervolt (2019), SGAxe (2020), ÆPIC Leak (2022)
- **SEV vulnerabilities**: SEVered (2018), ciphertext side channels (2021)
- **Pattern**: New attacks disclosed every 12-24 months; patched via microcode updates within 3-6 months

**Mitigation Progress**:
- SEV-SNP (2021+): Mitigates memory integrity attacks via RMP (Reverse Map Table)
- TDX (2023+): Designed post-Spectre with built-in side-channel protections
- **Industry consensus**: Side-channel risk declining as newer architectures incorporate defenses by design

**2. Trusted Computing Base (TCB) Size**
- Larger TCB = more code = greater attack surface
- **SGX TCB**: ~1MB (CPU microcode + enclave code)
- **TDX TCB**: ~10MB (firmware + virtualization stack)
- **SEV-SNP TCB**: ~50MB (includes hypervisor components)

**Implication**: SGX has smallest TCB but limited to small workloads. TDX offers best balance (small TCB + full VM isolation).

**3. Physical Access Attacks**
- TEEs assume physical security of hardware
- Advanced nation-state attackers with physical access can potentially extract keys via:
  - Laser fault injection
  - EM side-channel analysis
  - Supply chain interdiction

**Risk Level**: LOW for commercial use cases. HIGH for nation-state threat models (requires additional hardware security modules).

**4. Attestation Key Compromise**
- If manufacturer's root attestation key leaks, attackers could forge attestation reports
- **Historical incidents**: None publicly disclosed for Intel/AMD/NVIDIA
- **Mitigation**: Key rotation mechanisms, hardware-based key storage (fuses)
